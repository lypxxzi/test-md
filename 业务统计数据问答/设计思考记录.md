# 中间表设计思考记录

---

## Q1：宽表是什么？为什么选择宽表？

### 宽表的定义

宽表是把**一个业务主体**在**某个时间切面**上的**所有指标**，全部摊平成一行里的不同列的表结构。

对应到我们的场景：
- 业务主体 = 客户
- 时间切面 = 一个月
- 所有指标 = 订单量、订单额、发货、利润、回款、应收款...

一行就是这个客户这个月的"全貌"。

---

### 关键术语

| 术语 | 含义 | 在表里的角色 | 例子 |
|------|------|------------|------|
| **业务主体** | 这张表每一行在描述"谁" | 主键的一部分 | 表1的主体是"客户"，表2的主体是"客户+产品" |
| **维度** | 用来定位和筛选数据的字段，回答"在哪/什么时候/哪个类别" | 出现在 WHERE（筛选）和 GROUP BY（分组）里 | stat_month, province, product_category |
| **指标** | 被统计和度量的数字，回答"多少/多大/什么比率" | 出现在 SELECT 和聚合函数里（SUM/AVG） | order_count, actual_profit, on_time_count |
| **粒度** | 一行精确到什么程度，即哪些字段组合在一起能唯一确定一行 | 主键/唯一标识 | 表1 = 客户×月，表2 = 客户×产品×月 |

**三者关系：**
```
一行数据 = 业务主体 + 维度 + 一组指标

例：
客户张三（业务主体）+ 2025年3月（时间维度）+ 广东省（地域维度）
→ 订单量=15，订单额=80万，利润=12万（指标）
```

**维度和指标都是表里的字段（列）**，区别只在用途——维度用来筛选和分组，指标用来计算和展示。维度不是判断是否需要新表的条件。

**粒度和业务主体的关系：** 业务主体是粒度的一部分，但粒度还包含时间维度等。同一个业务主体（客户），按月存就是"客户×月"粒度，按天存就是"客户×天"粒度。决定是否需要新表的条件是**粒度是否不同**，不是维度多少。

---

### 宽表 vs 窄表对比

| | 窄表（传统关系型） | 宽表 |
|--|------------------|------|
| 结构 | 每个主题拆成单独的表，通过外键关联 | 一个主题的所有指标放在同一张表的不同列 |
| 查询 | 需要 JOIN 多张表才能拿到完整数据 | 单表查询，不需要 JOIN |
| 列数 | 每张表 5~15 列 | 一张表 30~300 列 |
| 适合的存储引擎 | MySQL、PostgreSQL（行存储） | ClickHouse、Doris（列存储） |

### 举例

**窄表方式**（传统做法）：
```
订单量表：tenant_id, customer_id, stat_month, order_count, shipped_count
利润表：  tenant_id, customer_id, stat_month, budget_cost, actual_profit
回款表：  tenant_id, customer_id, stat_month, collected, pending
```
→ 查"某客户本月订单量+利润+回款"需要 JOIN 三张表。

**宽表方式**（我们的做法）：
```
一张表：tenant_id, customer_id, stat_month, order_count, shipped_count, budget_cost, actual_profit, collected, pending, ...
```
→ 单表查询，WHERE customer_id = X 就拿到全部数据。

---

### 宽表的使用场景

宽表用在**数据仓库的服务层（DWS）**，面向分析查询，不是业务系统的增删改查：

| 场景 | 为什么适合宽表 |
|------|--------------|
| BI 报表/数据看板 | 一个页面要同时展示多个维度的指标，宽表一次查出来 |
| AI 数据问答 | 用户问的问题不可预测，可能涉及任何字段组合，宽表不用提前设计 JOIN 路径 |
| OLAP 多维分析 | 按时间/地域/产品类型做聚合分析，宽表单表搞定 |

**不适合**业务系统的事务操作（下单、改状态、扣库存），那些还是用传统的窄表+关系型数据库。

---

### 为什么宽表适合我们的场景

1. **ClickHouse 是列存储引擎**——查哪列读哪列，其他列不占 IO。38 列和 5 列的查询速度一样（只要你实际查的列数相同）。
2. **AI 问答场景需要灵活组合指标**——用户可能问"订单量+利润"，也可能问"回款+交付率"。宽表不管怎么组合都是单表查询，不用预判用户会问什么组合。
3. **避免 JOIN**——ClickHouse 的 JOIN 性能远不如单表扫描。宽表消灭了 JOIN 的需求。
4. **加字段成本低**——新需求来了，加一列就行，不用新建表和维护 JOIN 关系。

---

### 宽表的代价

| 代价 | 在我们场景下是否构成问题 |
|------|----------------------|
| 存储空间稍大 | 否。列存储压缩比高（10:1），且数据量不大（几十万~百万行级别） |
| 写入时要算更多字段 | 否。ETL 定时任务跑完即可，不是实时写入 |
| 某些字段可能为 NULL（稀疏） | 否。ClickHouse 对 NULL/默认值的存储几乎不占空间 |
| 字段权限控制变成列级别 | 通过应用层权限元数据表解决，不影响表结构 |

---

### 结论

**我们用宽表，因为：ClickHouse 列存储 + AI 灵活查询 + 无 JOIN + 扩展方便。代价在我们的场景下可以忽略。**

前提条件：底层是列存储引擎（ClickHouse），查 3 列就只读 3 列的数据文件，剩下的列不被读取，列再多也不影响查询速度。

## Q2：做 DWS 宽表必须遵循哪些建模规范？

以下规范都来自数据仓库建模的行业标准（Kimball 维度建模方法论 / 阿里 OneData / 字节、美团数仓规范），**不是我们自己的设计选择，而是所有正规数仓团队都遵循的通用规范**。

---

### 规范1：增量 vs 快照（DWS 日表的核心规则）

日粒度表中，不同类型的指标存法不同：

- **流量型指标**（订单量、订单额、利润等"发生了多少"）→ 存**当日增量**，不存累计
- **存量型指标**（未收款余额、在产订单数等"当前是多少"）→ 存**当日快照**

为什么是强制规范，不是可选项：

| 原因 | 说明 |
|------|------|
| 聚合灵活性 | 存增量可以 SUM 任意时间段（3天/7天/月/季）。存累计要算"本月新增"还得做减法，易错 |
| 数据可修复 | 某天算错只需重跑那一天。存累计的话改一天要级联修正后面所有天 |
| 行与行独立 | 每行只描述当天，不依赖前一行，补数/重跑不需要顺序 |
| 存储效率 | 累计值越来越大，增量值相对稳定，压缩比更好 |

字段示例（每个字段必须标注"增量/快照"）：
```
order_count          int      当日新增订单量（增量）
order_amount         decimal  当日新增订单金额（增量）
in_production_count  int      截至当日在产订单数（快照）
total_uncollected    decimal  截至当日合计未收款（快照）
```

---

### 规范2：粒度一致性

**一张表里每一行的粒度必须完全相同。**

错误做法（混了不同粒度）：
```
行1：客户A, 2025-03-05, 订单量=3     ← 一天的数据
行2：客户A, 2025-03,    订单量=45    ← 一个月的数据
```

正确做法：表里每行都是"一个客户×一天"。月数据通过查询时 SUM 得出，不混存在表里。

---

### 规范3：指标可加性标注

每个指标必须明确它能否跨时间/跨维度做 SUM：

| 类型 | 含义 | 能否 SUM | 例子 |
|------|------|---------|------|
| 可加指标 | 任何维度都能直接 SUM | 能 | 订单量、订单额、利润 |
| 半可加指标 | 能跨部分维度，不能跨时间 | 部分能 | 在产订单数（能跨客户，不能跨天） |
| 不可加指标 | 不能直接 SUM | 不能 | 利润率%、按期交付率%（比率类） |

**规则：宽表只存可加/半可加的原始数值，不存比率。** 比率查询时现算：
```sql
-- 正确：存 on_time_count 和 total_product_count，查询时算比率
SELECT on_time_count * 100.0 / total_product_count AS on_time_rate
-- 错误：直接存 on_time_rate = 85.5%（无法聚合，85% + 90% ≠ 175%）
```

---

### 规范4：金额用 Decimal，不用 Float

金额、成本、利润等财务字段必须用 `Decimal(14,2)` / `Decimal(18,2)`，不能用 Float/Double。Float 有精度丢失（0.1+0.2≠0.3），财务数据不允许。

---

### 规范5：主键唯一性

每张表必须有明确的复合主键，保证不出现重复行：
```
表1主键：tenant_id + stat_date + customer_id
表2主键：tenant_id + stat_date + customer_id + product_id
```
ETL 写入时同一主键不能出现两行。重跑某天数据是覆盖，不是追加。

---

### 规范6：幂等写入

定时任务重跑同一天的数据，结果必须和第一次一样——不产生重复，也不丢数据。

实现方式：先删后插（DELETE 当天 → INSERT 重算结果），或用 ReplacingMergeTree 引擎自动去重。

---

### 规范7：时间字段统一格式

- 日期统一用 `Date` 类型（如 `2025-03-05`），不用字符串
- 时区统一（全部 UTC 或全部业务时区，不能混）
- 命名统一：`stat_date`（日粒度）/ `stat_month`（月粒度），不要一会儿 date 一会儿 dt

---

### 规范8：NULL 处理

| 场景 | 处理方式 |
|------|---------|
| 当天该客户没有订单 | 数值字段填 0，不填 NULL |
| 该客户确实没有某类数据 | 可不生成该行，或填 0 |
| 维度字段值未知 | 填 '未知'/'unknown'，不填 NULL |

原因：ClickHouse 聚合函数对 NULL 有坑（SUM 跳过 NULL 行），用 0 更安全。

---

### 规范9：分区策略

ClickHouse 表必须设置分区，通常按月：
```sql
PARTITION BY toYYYYMM(stat_date)
```
好处：查某月只扫该分区，删历史数据按分区整块删，不影响其他数据。

---

### 规范10：租户字段在首位

`tenant_id` 放表的第一个字段 + ORDER BY 第一位。几乎所有查询都带 tenant_id 过滤，放第一位索引命中率最高。

---

### 规范11：维度退化（Degenerate Dimension）

**宽表不能只存 ID，必须把对应的名称、分类等维度属性一起平铺进来。**

| 错误做法 | 正确做法 |
|---------|---------|
| 只存 customer_id | 同时存 customer_id + customer_name |
| 只存 product_id | 同时存 product_id + product_name + product_category |
| 查询时 JOIN 维表拿名称 | 不 JOIN，宽表里直接有 |

原因：宽表的核心目的就是消灭 JOIN；ClickHouse JOIN 性能差；AI 问答需要直接看到可读名称而非 ID。维度属性冗余存储是刻意为之，用空间换掉 JOIN。

---

### 规范12：命名规范

| 层面 | 规则 | 例子 |
|------|------|------|
| 表名 | dws_主题_粒度 | dws_customer_daily |
| 字段名 | 全小写 + 下划线（snake_case） | order_count, actual_profit |
| 指标字段 | 动作/状态 + 度量类型 | shipped_amount, overdue_count |
| 维度字段 | 实体 + 属性 | customer_name, product_category |
| 布尔/标记字段 | is_ / has_ 前缀 | is_active, has_order |

所有表统一，不能一会儿驼峰一会儿下划线。

---

### 规范13：排序键设计（ORDER BY）

ClickHouse 的 ORDER BY 是**数据在磁盘上的物理存储顺序**，直接决定查询性能：
```sql
ORDER BY (tenant_id, stat_date, customer_id)
```
规则：最常出现在 WHERE 的字段放前面；基数从低到高（tenant_id 低 → stat_date 中 → customer_id 高）；选错会导致全表扫描。

---

### 规范14：缓慢变化维度处理（SCD）

维度属性会变（客户改名、换城市、产品重分类），处理方式：

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| 直接覆盖（SCD Type 1） | 新数据用最新名称/分类，历史行不改 | 不需追溯历史变化（常用） |
| 保留历史（SCD Type 2） | 维度变了生成新行，旧行保留旧值 | 需要"当时"的状态 |

大多数 DWS 宽表用 Type 1。需要追溯时在 DWD 层处理。

---

### 规范15：数据生命周期（TTL）

必须定义数据保留多久，到期自动清理：
```sql
TTL stat_date + INTERVAL 3 YEAR
```
DWS 日表通常保留 2~3 年，超期自动删除或归档冷存储，不能无限增长。

---

### 规范16：字段分组有序排列

字段按逻辑分组排列，不能乱序：
```
1. 维度字段（tenant_id, stat_date, customer_id, customer_name...）
2. 状态标记字段（is_active, has_order...）
3. 业务指标字段-按主题分组（订单类 → 发货类 → 利润类 → 回款类）
```
方便阅读、维护和知道新字段加在哪。

---

### 规范17：数据血缘标注

每个字段必须标注数据来源——从哪张源表、什么计算逻辑得来：
```
order_count:   来源=ods_order, 逻辑=COUNT(order_id) WHERE order_date = stat_date
actual_profit: 来源=ods_order_cost, 逻辑=SUM(revenue - actual_cost) WHERE shipped_date = stat_date
```
不标注的后果：字段含义不清，后续维护者不知道数字怎么算的，出问题无法追溯。

---

### 规范汇总清单

| # | 规范 | 一句话 |
|---|------|--------|
| 1 | 增量/快照区分 | 流量存当日增量，存量存当日快照 |
| 2 | 粒度一致性 | 一张表每行粒度必须相同 |
| 3 | 可加性标注 | 只存原始数值，比率查询时现算 |
| 4 | Decimal 精度 | 金额类用 Decimal，不用 Float |
| 5 | 主键唯一性 | 复合主键唯一标识每一行 |
| 6 | 幂等写入 | 重跑不产生重复数据 |
| 7 | 时间格式统一 | Date 类型 + 统一时区 + 统一命名 |
| 8 | NULL 处理 | 数值填 0，维度填 '未知' |
| 9 | 分区策略 | 按月分区 |
| 10 | 租户字段在首位 | tenant_id 做 ORDER BY 第一列 |
| 11 | 维度退化 | 不存裸 ID，平铺名称/分类进宽表 |
| 12 | 命名规范 | snake_case，表名=dws_主题_粒度 |
| 13 | 排序键设计 | 按查询频率+基数从低到高排 ORDER BY |
| 14 | 缓慢变化维度 | 明确选择覆盖还是保留历史 |
| 15 | 数据生命周期 | 定义 TTL，到期自动清理 |

## 待梳理问题：月度数据会被后续冲销/调整影响

业务人员如果只看某个月的汇总数据，可能会把“当月发生的业绩”误认为“最终有效业绩”。

典型风险：
- 后续月份发生退货、冲销、红冲、调整时，历史月份的最终结果会变化；
- 单独看某个月的汇总值，无法判断它是当月发生值，还是调整后的有效值；
- 如果业务把未区分口径的数据用于考核、复盘、财务统计，可能长期用错数字。

后续需要重点梳理：
- 月度表里是否要区分“当月发生额”和“调整后净额”；
- 冲销/调整应该归属到原业务月份，还是发生调整的月份；
- 用户问“某月业绩多少”时，AI 默认应该回答哪个口径，或是否同时展示多个口径。

## 待梳理问题：日表和月表的设计边界

<span style="color:red">当前需要进一步明确：日表和月表应该如何设计？月表是按具体业务主体来做，还是按公司/租户整体来做？</span>

初步问题点：
- 日表是否作为基础表，用来支撑近30天、近90天、近180天、近一年等滚动统计；
- 月表是否作为汇总表，用来支撑按月、季度、年度的报表和趋势；
- 月表的统计对象需要明确：是客户、客户+产品、产品、业务员、地区，还是公司/租户整体；
- 是否需要同时存在“业务主体月表”和“公司整体月表”，分别回答不同层级的问题。
